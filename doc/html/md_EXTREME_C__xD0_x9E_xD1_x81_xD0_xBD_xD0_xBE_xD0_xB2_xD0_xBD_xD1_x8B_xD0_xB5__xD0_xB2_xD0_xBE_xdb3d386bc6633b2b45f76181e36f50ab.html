Lab 02: Макросы и управление потоком выполнения в C

Репозиторий содержит комплект из трех проверочных работ и тестов к ним. Материалы проверяют навыки работы с препроцессором, синтаксисом макросов, заменяющих управляющие конструкции, и понимание побочных эффектов макроподстановок.

📅 Проверочная работа №1: Развёртывание макросов и препроцессинг

Цель: Проверить понимание механики текстовой подстановки макросов LOOP, PRINT и ENDLOOP до этапа компиляции.

Задание (Практика)

  1. Выполните ручное развёртывание препроцессора для функции main. Напишите чистый код на C, который получится после того, как препроцессор заменит макросы LOOP, PRINT и ENDLOOP.
  2. Объясните, почему синтаксис PRINT(counter) в исходном коде не имеет точки с запятой ; на конце, но код при этом компилируется без ошибок.
  3. Опишите, в какой момент времени (на этапе препроцессинга, компиляции или выполнения) макросы прекращают свое существование в кодовой базе.

Тест №1

  1. Во что превратится строка PRINT(counter) после работы препроцессора?
    • А) printf("%d\n", counter)
    • Б) printf("%d\n", counter);
    • В) PRINT(10);
    • Г) Вызов функции PRINT с аргументом counter.
  2. Какая директива препроцессора используется для отмены определения макроса (например, чтобы отключить LOOP)?
    • А) #undef
    • Б) #delete
    • В) #clear
    • Г) #ifndef
  3. Что произойдет, если случайно написать две точки с запятой: PRINT(counter); внутри данного цикла?
    • А) Ошибка компиляции (Syntax Error).
    • Б) Ошибка времени выполнения (Runtime Error).
    • В) Код скомпилируется успешно, так как вторая точка с запятой интерпретируется как пустой оператор.
    • Г) Цикл выполнится только один раз.

🔀 Проверочная работа №2: Область видимости и побочные эффекты

Цель: Изучить концепцию области видимости переменных, объявленных внутри скрытых макросами конструкций, и разобрать проблему побочных эффектов.

Задание (Практика)

  1. Какова область видимости переменной counter, созданной макросом LOOP(counter, 1, 10)? Можно ли обратиться к переменной counter сразу после строки ENDLOOP (например, вызвать printf("%d", counter);)? Обоснуйте ответ.
  2. Предположим, программист вызвал макрос следующим образом: PRINT(counter++). Опишите, к какому побочному эффекту (side effect) это приведет и сколько раз в реальности увеличится переменная при каждом вызове макроса.

Тест №2

  1. Где будет доступна переменная v, объявленная внутри макроса #define LOOP(v, s, e) for (int v = s; v <= e; v++) {?
    • А) Во всей функции main.
    • Б) Только внутри блока тела цикла for, ограниченного фигурными скобками { и }.
    • В) Во всем файле (глобальная область видимости).
    • Г) Переменная уничтожается сразу после вызова LOOP.
  2. Что произойдет, если передать выражение с побочным эффектом в макрос #define SQUARE(x) ((x) * (x)) в виде SQUARE(a++)?
    • А) Выражение a++ выполнится один раз.
    • Б) Переменная a увеличится дважды, так как x подставляется в макрос в два разных места.
    • В) Произойдет ошибка сегментации.
    • Г) Макрос автоматически запретит инкремент.
  3. Какое ключевое отличие макроса от обычной inline-функции в языке C?
    • А) Inline-функции работают медленнее макросов.
    • Б) Макросы не проверяют типы данных аргументов и являются чисто текстовой подстановкой.
    • В) Inline-функции обрабатываются препроцессором.
    • Г) Макросы могут возвращать значения через оператор return.

💾 Проверочная работа №3: Безопасность макросов и рефакторинг

Цель: Освоить правила безопасного написания макросов и переписать код с использованием классических идиом программирования на C.

Задание (Практика)

  1. Макрос PRINT(a) написан небезопасно, так как аргумент a не обернут в круглые скобки. Напишите правильный, безопасный вариант объявления этого макроса, защищающий от неверного порядка вычислений при передаче выражений (например, PRINT(5 + 5)).
  2. Проведите рефакторинг исходного кода: полностью избавьтесь от макросов LOOP, PRINT и ENDLOOP. Перепишите программу, используя стандартный цикл for и функцию printf, оформленную в виде отдельной статической функции static void print_int(int value).

Тест №3

  1. Почему в сложных макросах все аргументы и само выражение рекомендуется оборачивать в круглые скобки?
    • А) Без скобок макросы не компилируются в стандарте C99.
    • Б) Чтобы гарантировать правильный приоритет выполнения операций при подстановке сложных выражений.
    • В) Для автоматического приведения типов.
    • Г) Чтобы выделить макрос в памяти стека.
  2. Какая стандартная идиома (конструкция) используется в C для создания многострочных безопасных макросов, объединяющих несколько команд?
    • А) if (1) { ... }
    • Б) switch (0) { ... }
    • В) do { ... } while (0)
    • Г) for (;;) { ... }
  3. Какое значение вернет программа main операционной системе, если цикл LOOP отработает штатно?
    • А) 10
    • Б) 0
    • В) 1
    • Г) Количество итераций цикла.

🔑 Ответы и разборы тестов

Разбор Теста №1

  1. Правильный ответ: Б. Макрос объявлен как printf("%d\n", a);. Символ ; уже включен в тело макроса, поэтому строка развернется вместе с точкой с запятой.
  2. Правильный ответ: А. Директива #undef удаляет текущее определение макроса, позволяя препроцессору забыть его или переопределить заново.
  3. Правильный ответ: В. Две точки с запятой подряд образуют пустой оператор ;. В языке C пустой оператор является валидным и не вызывает ошибок компиляции в данном контексте.

Разбор Теста №2

  1. Правильный ответ: Б. Поскольку переменная объявлена как int v прямо внутри заголовка цикла for (int v = ...), ее область видимости (scope) ограничена телом этого цикла.
  2. Правильный ответ: Б. Макрос выполняет буквальную подстановку текста. Выражение превратится в ((a++) * (a++)), из-за чего инкремент сработает дважды, что также приводит к неопределенному поведению (Undefined Behavior).
  3. Правильный ответ: Б. Препроцессор ничего не знает о типах данных (int, char и т.д.). Он лишь подставляет текст, тогда как inline-функции являются полноценными функциями с проверкой типов компилятором.

Разбор Теста №3

  1. Правильный ответ: Б. Приоритет макроподстановки может сломать математическую логику. Например, если #define MULT(x) x * x, то MULT(2 + 2) превратится в 2 + 2 * 2 + 2 = 8, вместо ожидаемых 16. Скобки ((x) * (x)) решают эту проблему.
  2. Правильный ответ: В. Конструкция do { ... } while(0) группирует несколько инструкций в один блок, который ведет себя как единый оператор и корректно работает внутри конструкций if-else без риска сломать синтаксис.
  3. Правильный ответ: Б. В конце функции main явно прописана инструкция return 0;, что сигнализирует операционной системе об успешном завершении программы.

🛠️ Эталонные решения практических заданий

Решение работы №1 (Развёртывание кода)

  1. Чистый C-код после этапа препроцессинга:
    #include <stdio.h>
    int main(int args, char **argv) {
    for (int counter = 1; counter <= 10; counter++) {
    printf("%d\n", counter);
    }
    return 0;
    }
  2. Отсутствие точки с запятой: Она уже заложена в самом определении макроса: #define PRINT(a) printf("%d\n", a);. Подставляя макрос, препроцессор переносит и знак ;.
  3. Время жизни макросов: Макросы перестают существовать сразу после этапа препроцессинга. Компилятор получает на вход код, где макросов уже нет — вместо них находится чистый C-код.

Решение работы №2 (Область видимости и побочные эффекты)

  1. Область видимости: Обратиться к counter после ENDLOOP нельзя. Будет вызвана ошибка компиляции ‘error: 'counter’ undeclared. Переменная уничтожается при выходе из блока циклаfor.
  2. **Побочный эффектPRINT(counter++):** Строка развернется вprintf("%d\n", counter++);. В данном конкретном случае переменная увеличится **один раз** (так как аргументaупоминается в теле макроса один раз). Однако значение сначала передастся в функцию, а затем произойдет инкремент, что нарушит логику шага циклаcounter++в самомfor`.

Решение работы №3 (Рефакторинг и безопасный макрос)

  1. Безопасная версия макроса:
    #define PRINT(a) printf("%d\n", (a));
  2. Рефакторинг исходного кода (clean code без макросов):
    #include <stdio.h>
    // Статическая функция для вывода числа
    static void print_int(int value) {
    printf("%d\n", value);
    }
    int main(int args, char **argv) {
    // Использование стандартного цикла for
    for (int counter = 1; counter <= 10; counter++) {
    print_int(counter);
    }
    return 0;
    }

🛠️ Руководство: Как посмотреть результат работы препроцессора (флаг <tt>gcc -E</tt>)

Поскольку макросы LOOP, PRINT и ENDLOOP являются лишь инструкциями для препроцессора, компилятор их не видит. Чтобы понять, во что превращается ваш код до этапа компиляции, вы можете заставить gcc остановиться сразу после шага препроцессинга.

Способ 1. Использование Makefile (Рекомендуемый)

Если в проекте присутствует Makefile, просто выполните в терминале команду:

make preprocess

После этого в корне проекта появится файл main.i, содержащий развёрнутый код.

Способ 2. Ручной запуск через терминал

Вы можете вызвать препроцессор напрямую с помощью флага -E:

gcc -E main.c -o main.i

Как анализировать файл <tt>main.i</tt>:

  1. Откройте сгенерированный файл main.i в текстовом редакторе.
  2. В нём будет много строк (около нескольких сотен или тысяч) — это подключенный заголовочный файл <stdio.h>, который препроцессор скопировал целиком вместо строки #include.
  3. Прокрутите файл в самый конец. Там вы увидите вашу функцию main, в которой макросы заменены на стандартные конструкции языка C:
int main(int args, char **argv) {
for (int counter = 1; counter <= 10; counter++) {
printf("%d\n", counter);
}
return 0;
}

Изучение файла .i помогает наглядно увидеть, почему макросы называют «умной автозаменой текста» и как они могут скрывать синтаксические ошибки.
