# 📝 Проверочные работы и тесты: Валидация аргументов и вариативные макросы

Данный методический материал содержит **3 проверочные работы** и **тестовые задания** к ним для контроля усвоения тем: работа препроцессора Си, обработка параметров командной строки (`argc`/`argv`) и проектирование вариативных макросов (`__VA_ARGS__`).

---

## 🛠️ Проверочная работа №1: Базовый уровень
**Тема**: Синтаксис вариативных макросов и потоки вывода

### 📋 Задание
Изучите структуру макроса из предоставленного текста:
```c
#define LOG_ERROR(format, ...) \
    fprintf(stderr, format, __VA_ARGS__)
```
1. Опишите своими словами, в чем заключается главное архитектурное различие между потоками вывода `stdout` (куда пишет `printf`) и `stderr` (куда пишет `LOG_ERROR`). Почему системные ошибки важно отправлять именно в `stderr`?
2. Напишите, во что преобразуется строка `LOG_ERROR("Code %d: %s", 404, "Not Found");` после обработки препроцессором.
3. Проанализируйте вызов `LOG_ERROR("Connection lost.");`. Объясните, почему этот код вызовет синтаксическую ошибку в строгом стандарте **C99**.

### ✏️ Тест к проверочной работе №1
1. **Какое ключевое слово препроцессора отвечает за подстановку всех аргументов, переданных в макрос через троеточие?**
   * А) `__VARIADIC__`
   * Б) `__VA_ARGS__`
   * В) `__MACRO_ARGS__`

2. **Куда физически перенаправляет текстовые данные функция `fprintf(stderr, ...)` по умолчанию?**
   * А) В стандартный поток вывода ошибок (обычно выводится на экран терминала).
   * Б) Автоматически записывает в текстовый файл `error.log` на диске.
   * В) В буферизированный поток стандартного вывода приложения.

3. **Какая фаза сборки программы отвечает за развертывание макроса `LOG_ERROR`?**
   * А) Линковка (компоновка)
   * Б) Препроцессинг
   * В) Генерация ассемблерного кода

## 🚀 Проверочная работа №2: Средний уровень
**Тема**: Валидация параметров командной строки `argc` и `argv`

### 📋 Задание
Перед вами фрагмент защитного барьера функции `main`:
```c
if (argc < 3) {
    LOG_ERROR("Invalid number of arguments for version %s\n.", VERSION);
    exit(1);
}
```
1. Допустим, пользователь запустил программу из терминала следующей командой: `./app -n`. Напишите точные значения переменных `argc` и `argv[0]`. Пройдет ли программа данный защитный барьер?
2. Объясните, к каким последствиям с точки зрения управления памятью привел бы пропуск этой проверки, если бы далее в коде сразу шло обращение к элементам `argv[1]` и `argv[2]`.
3. Какую роль выполняет системная функция `exit(1)`? Чем её поведение отличается от стандартного оператора `return 0` внутри функции `main`?

### ✏️ Тест к проверочной работе №2
1. **Что гарантированно содержит в себе элемент `argv[0]` при запуске консольного приложения?**
   * А) Первый флаг управления, переданный пользователем.
   * Б) Общее количество аргументов командной строки.
   * В) Путь к исполняемому файлу или имя запускаемой программы.

2. **Если программа вызвана командой `./my_program -n 125`, какое число запишется в переменную `argc`?**
   * А) 2
   * Б) 3
   * В) 4

3. **Какая критическая ошибка произойдет в рантайме, если попытаться прочитать `argv[5]`, когда значение `argc` равно 2?**
   * А) Ошибка компиляции `undefined reference`.
   * Б) Выход за границы массива / чтение чужой памяти (`Out-of-Bounds Read`).
   * В) Переполнение стека (`Stack Overflow`).

## 💎 Проверочная работа №3: Профессиональный уровень
**Тема**: Строковая валидация и эволюция стандартов макросов

### 📋 Задание
В коде выполняется проверка флага:
```c
if (strcmp(argv[1], "-n") != 0) {
    LOG_ERROR("%s is a wrong param at index %d for version %s.", argv[1], 1, VERSION);
    exit(1);
}
```
1. Почему для проверки равенства строк используется функция `strcmp(argv[1], "-n") != 0`, а не классическое сравнение через операторы равенства `argv[1] == "-n"`? Что именно сравнивал бы оператор `==`?
2. Модифицируйте макрос `LOG_ERROR` из текста с помощью нестандартного расширения GNU (`##__VA_ARGS__`), чтобы он поддерживал вызов сообщений без дополнительных параметров (например, `LOG_ERROR("Critical error!");`), не вызывая синтаксических ошибок из-за висящей запятой.
3. Опишите, как современный стандарт **C23** решает проблему пустых вариативных аргументов с помощью конструкции `__VA_OPT__`. Напишите пример макроса `LOG_ERROR` с использованием `__VA_OPT__`.

### ✏️ Тест к проверочной работе №3
1. **Что возвращает функция `strcmp`, если сравниваемые строки абсолютно идентичны?**
   * А) 1
   * Б) Округленный размер строк
   * В) 0

2. **В чем заключается главная опасность уязвимости «Use-After-Free» или некорректного завершения программы при ошибках валидации данных?**
   * А) Программа начинает работать медленнее.
   * Б) Происходит утечка дескрипторов или эксплуатация поврежденного состояния памяти процесса.
   * В) Компилятор удаляет исполняемый файл.

3. **Для чего в теле макроса Си строки форматирования часто объединяют с метаданными (например, `__FILE__`, `__LINE__`)?**
   * А) Для автоматического вычисления размера исполняемого файла.
   * Б) Для обогащения логов информацией о точном месте возникновения ошибки в исходном коде.
   * В) Это синтаксическое требование функции `fprintf`.

## 🔑 Ответы и критерии оценивания для преподавателя

### Ответы на тестовые задания
* **Тест №1**: 1-Б, 2-А, 3-Б
* **Тест №2**: 1-В, 2-Б, 3-Б
* **Тест №3**: 1-В, 2-Б, 3-Б

### Эталоны ответов на практические задания

#### Проверочная работа №1
1. `stdout` буферизируется операционной системой (вывод может задерживаться), а `stderr` — не буферизируется (данные выводятся мгновенно). Ошибки пишут в `stderr`, чтобы они не потерялись при падении программы и их можно было легко отделить от полезного вывода приложения при перенаправлении потоков (например, через `> log.txt`).
2. Превратится в: `fprintf(stderr, "Code %d: %s", 404, "Not Found");`.
3. Вызов развернется в `fprintf(stderr, "Connection lost.", );`. Наличие висящей запятой перед закрывающей скобкой при пустом `__VA_ARGS__` является грубым нарушением синтаксиса **C99** и вызовет ошибку компиляции.

#### Проверочная работа №2
1. Значение `argc` будет равно `2`, а `argv[0]` запишет строку `"./app"`. Программа **не пройдет** защитный барьер, так как условие `argc < 3` (`2 < 3`) окажется истинным, и приложение аварийно завершится.
2. Произошло бы обращение к `argv[2]`, который в данном сценарии является нулевым указателем (`NULL`). Попытка разыменования или чтения данных из `NULL` приводит к падению программы с ошибкой `Segmentation Fault`.
3. Функция `exit(1)` немедленно останавливает весь процесс и возвращает ОС код закрытия `1` (признак ошибки) из любой точки приложения. Оператор `return 0` завершает выполнение текущей функции; в `main` он также закроет программу, но возвратит код `0` (успех).

#### Проверочная работа №3
1. Оператор `==` сравнивает адреса указателей в памяти (сравнивает, указывают ли `argv[1]` и строковый литерал на одну и ту же ячейку). Функция `strcmp` посимвольно сравнивает содержимое самих строк.
2. Решение через GNU-расширение: `#define LOG_ERROR(format, ...) fprintf(stderr, format, ##__VA_ARGS__)`.
3. Оператор `__VA_OPT__(contents)` генерирует свое содержимое только если список `__VA_ARGS__` не пуст. Пример для **C23**: `#define LOG_ERROR(format, ...) fprintf(stderr, format __VA_OPT__(,) __VA_ARGS__)`.

* **Хорошо (4)**: Студент успешно справился с логикой валидации `argc`/`argv`, но допустил незначительные ошибки в механике раскрытия макросов компилятором.
