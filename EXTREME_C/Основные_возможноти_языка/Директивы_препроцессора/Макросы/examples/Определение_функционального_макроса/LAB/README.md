# 📝 Проверочные работы и тесты: Функциональные макросы Си и защита приоритетов

Данный методический материал содержит **3 проверочные работы** и **тестовые задания** к ним для контроля усвоения тем: синтаксис функциональных макросов (макросов с параметрами), защита приоритетов операций с помощью экранирующих скобок, побочные эффекты макроподстановок и подавление предупреждений компилятора.

---

## 🛠️ Проверочная работа №1: Базовый уровень
**Тема**: Текстовое развертывание функциональных макросов

### 📋 Задание
Изучите структуру макроса из предоставленного текста:
```c
#define ADD(a, b) ((a) + (b))
```
1. Опишите своими словами механизм подстановки аргументов в функциональный макрос на этапе препроцессинга. Чем этот процесс принципиально отличается от передачи аргументов в обычную функцию (например, `int add(int a, int b)`)?
2. Напишите, во что превратится строка `int value = ADD(10, 20);` после прохождения фазы препроцессинга.
3. Какую роль выполняют конструкции вида `(void)z;` в функции `main`? Какие предупреждения компилятора они подавляют при сборке со строгими флагами?

### ✏️ Тест к проверочной работе №1
1. **Какая фаза сборки программы отвечает за прямую текстовую замену вызова `ADD(x, y)` на арифметическое выражение?**
   * А) Линковка
   * Б) Препроцессинг
   * В) Компиляция

2. **Происходит ли проверка типов данных аргументов (`int`, `float`, `char`) в момент развертывания макроса `ADD` препроцессором?**
   * А) Да, препроцессор проверяет типы данных.
   * Б) Нет, препроцессор выполняет исключительно текстовую замену, типы проверит компилятор позже.
   * В) Проверка типов происходит только для вещественных чисел.

3. **Что делает конструкция `(void)argc;` в кодовой базе?**
   * А) Обнуляет значение переменной `argc`.
   * Б) Удаляет переменную из оперативной памяти.
   * В) Явно сообщает компилятору, что переменная создана намеренно, подавляя варнинг о её неиспользовании.

## 🚀 Проверочная работа №2: Средний уровень
**Тема**: Защитное экранирование параметров и приоритеты операций

### 📋 Задание
Представьте, что разработчик допустил ошибку и объявил макрос сложения без защитных скобок:
```c
#define ADD_BAD(a, b) a + b
```
1. Рассмотрите выражение `int result = ADD_BAD(5 & 1, 2 | 4);`. Пользуясь таблицей приоритетов операторов Си (где сложение `+` имеет более высокий приоритет, чем побитовые операции `&` и `|`), распишите пошагово, в каком порядке компилятор выполнит действия в этой строке.
2. Напишите, почему Вариант с защитными скобками `((5 & 1) + (2 | 4))` от макроса `ADD(a, b)` отработает математически корректно. Что именно гарантирует атомарность каждого операнда?
3. Во что препроцессор развернет вызов `int total = ADD_BAD(10, 20) * 2;`? Будет ли результат соответствовать математическому ожиданию пользователя `(10 + 20) * 2 = 60`?

### ✏️ Тест к проверочной работе №2
1. **Каковы будут последствия вызова `int total = ADD_BAD(10, 20) * 2;` из-за отсутствия внешних скобок у макроса?**
   * А) Код развернется в `10 + 20 * 2`, результат будет равен `50` вместо `60`.
   * Б) Компилятор выдаст синтаксическую ошибку `Invalid expression`.
   * В) Произойдет переполнение стека в рантайме.

2. **Зачем нужны внутренние скобки вокруг каждого отдельного аргумента, например `(a)` и `(b)`?**
   * А) Чтобы превратить аргументы в строковые литералы.
   * Б) Чтобы защитить приоритет внутренних операций, если в макрос передано комплексное бинарное выражение.
   * В) Это требование стандарта для выделения памяти.

3. **Какое расширение файла содержит чистый Си-код с уже развернутыми скобками макроса `ADD`, но до генерации ассемблерных команд?**
   * А) `.o`
   * Б) `.s`
   * В) `.i`

## 💎 Проверочная работа №3: Профессиональный уровень
**Тема**: Побочные эффекты макроподстановок (Инкременты)

### 📋 Задание
Одной из главных опасностей функциональных макросов является повторное вычисление аргументов с побочными эффектами. Анализируется макрос:
```c
#define ADD(a, b) ((a) + (b))
```
1. Изучите следующий фрагмент кода:
   ```c
   int x = 2;
   int y = 5;
   int z = ADD(x++, y);
   ```
   Распишите, во что препроцессор развернет третью строчку. Сколько раз выполнится операция инкремента `x++`? Какое значение запишется в переменную `z` и чему будет равен `x` после завершения этой строки?
2. Теперь изучите макрос вычисления квадрата числа: `#define SQUARE(x) ((x) * (x))`. Распишите пошагово результат выполнения выражения `int res = SQUARE(i++);`, если изначально `i = 3`. Изменится ли количество итераций инкремента по сравнению с макросом `ADD`? Почему?
3. Предложите современную и безопасную альтернативу макросу `ADD(a, b)` на чистом языке Си (начиная со стандарта C99), которая полностью исключает побочные эффекты повторного вычисления аргументов и имеет строгий контроль типов.

### ✏️ Тест к проверочной работе №3
1. **Сколько раз выполнится операция инкремента `i++` при развертывании макроса `SQUARE(i++)` в строку `((i++) * (i++))`?**
   * А) 1 раз, как в обычной функции.
   * Б) 2 раза, так как токен `i++` продублирован в теле макроса дважды.
   * В) Компилятор выдаст ошибку на этапе сборки.

2. **Что такое «точки следования» (Sequence Points) применительно к выражению `((i++) * (i++))`?**
   * А) Места в коде, где препроцессор обязан вставить комментарии.
   * Б) Границы, внутри которых многократное изменение одной переменной в рамках одного выражения вызывает неопределенное поведение (`Undefined Behavior`).
   * В) Точки останова отладчика `GDB`.

3. **Какое решение позволяет создать безопасную инлайновую замену кода Си без дефектов макросов?**
   * А) Использование статических глобальных переменных.
   * Б) Объявление функции с ключевым словом `static inline`.
   * В) Отказ от использования функций в проекте.

## 🔑 Ответы и критерии оценивания для преподавателя

### Ответы на тестовые задания
* **Тест №1**: 1-Б, 2-Б, 3-В
* **Тест №2**: 1-А, 2-Б, 3-В
* **Тест №3**: 1-Б, 2-Б, 3-Б

### Эталоны ответов на практические задания

#### Проверочная работа №1
1. Механизм функционального макроса выполняет **чисто текстовую замену** параметров на этапе препроцессинга (компилятор видит уже подставленный текст). Обычная функция передает значения через стек или регистры процессора в рантайме, имеет строгую типизацию аргументов и накладные расходы на вызов.
2. Превратится в: `int value = ((10) + (20));`.
3. Конструкции `(void)` явно указывают компилятору на намеренное неиспользование переменных. Они подавляют предупреждения `-Wunused-variable` и `-Wunused-parameter` при сборке со строгими флагами безопасности (`-Wall -Wextra -Werror`).

#### Проверочная работа №2
1. Строка развернется в `int result = 5 & 1 + 2 | 4;`. Согласно приоритетам Си: сначала выполнится сложение `1 + 2 = 3`. Затем выражение примет вид `5 & 3 | 4`. Побитовое И (`&`) приоритетнее, чем ИЛИ (`|`), поэтому выполнится `5 & 3 = 1`. И в конце выполнится `1 | 4 = 5`. Итоговое значение `5` не совпадает с математически ожидаемым `7`.
2. Вариант `((5 & 1) + (2 | 4))` отработает верно, так как внешние и внутренние скобки принудительно изолируют логические блоки, заставляя компилятор сначала вычислить `5 & 1 = 1` и `2 | 4 = 6`, а затем сложить результаты: `1 + 6 = 7`.
3. Вызов развернется в `int total = 10 + 20 * 2;`. Из-за приоритета умножения выполнится `20 * 2 = 40`, а затем `10 + 40 = 50`. Результат **не будет** равен 60 из-за отсутствия внешних скобок у макроса `ADD_BAD`.

#### Проверочная работа №3
1. Строка развернется в `int z = ((x++) + (y));`. Операция инкремента `x++` выполнится ровно **1 раз** (так как токен `a` встречается в теле макроса один раз). В переменную `z` запишется старое значение `x` плюс `y` (`2 + 5 = 7`). После завершения строки значение переменной `x` станет равно `3`.
2. Строка развернется в `int res = ((i++) * (i++));`. В данном случае токен `x` дублируется в теле макроса, поэтому инкремент выполнится **2 раза**. Это приводит к нарушению точек следования, неопределенному поведению (`UB`) и непредсказуемому результату вычислений.
3. Безопасная альтернатива — использование **`static inline`** функций:
   ```c
   static inline int add(int a, int b) {
       return a + b;
   }
   ```
   Она компилируется инлайново (без накладных расходов), проверяет типы и вычисляет аргумент с инкрементом ровно один раз до входа в тело функции.

### 📊 Критерии оценивания
* **Отлично (5)**: Практические задачи расписаны с пошаговым разбором приоритетов и препроцессинга. Тесты пройдены без ошибок. Продемонстрировано четкое понимание побочных эффектов инкрементов.
* **Хорошо (4)**: Верно определены результаты вычислений, но допущены неточности при описании механики точек следования (`Sequence Points`) или принципов работы `static inline`.
* **Удовлетворительно (3)**: Студент понимает базовую математику сложения, но путает текстовый препроцессинг с логикой выполнения кода в рантайме.
