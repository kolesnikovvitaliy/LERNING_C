# 📝 Проверочные работы и тесты: Механизм макроподстановок препроцессора Си

Данный методический материал содержит **3 проверочные работы** и **тестовые задания** к ним для контроля усвоения тем: фазы компиляции, текстовая подстановка директив `#define`, отличия макросов от типизированных переменных и анализ промежуточного кода приложения.

---

## 🛠️ Проверочная работа №1: Базовый уровень
**Тема**: Понятие препроцессинга и текстовая замена

### 📋 Задание
Изучите структуру демонстрационного макроса из предоставленного текста:
```c
#define ABC 5
```
1. Опишите своими словами, что означает утверждение: *«Макросы выполняют исключительно текстовую замену и не имеют типов»*. К каким последствиям это может привести при ошибочном объявлении макроса (например, `#define ABC 5.5.5`)?
2. Напишите, во что превратится строка `int val = ABC * 2;` после прохождения фазы препроцессинга.
3. Какая системная утилита в составе компилятора GCC отвечает за обработку директив, начинающихся с символа `#`?

### ✏️ Тест к проверочной работе №1
1. **Какая фаза сборки программы полностью удаляет идентификатор `ABC` из исходного кода, заменяя его значением?**
   * А) Линковка (компоновка)
   * Б) Препроцессинг
   * В) Ассемблирование

2. **Сколько памяти в байтах занимает макрос `#define ABC 5` в RAM устройства на этапе выполнения (рантайма) программы?**
   * А) 0 байт (не занимает места в памяти)
   * Б) 4 байта (размер стандартного int)
   * В) 1 байт (размер ASCII-символа)

3. **Что произойдет, если в коде написать `ABC = 10;`?**
   * А) Макрос динамически перезапишется новым значением.
   * Б) Ошибка препроцессора на этапе развертывания.
   * В) Ошибка компилятора, так как код превратится в невалидное выражение `5 = 10;`.

## 🚀 Проверочная работа №2: Средний уровень
**Тема**: Пошаговый разбор логики и побочные эффекты макросов

### 📋 Задание
Перед вами фрагмент исследуемой функции:
```c
int demo_macro_substitution_in_main(int argc, char **argv)
{
    int x = 2;
    int y = ABC;

    int z = x + y;
    return 0;
}
```
1. Распишите пошагово математическую логику вычисления переменной `z`, подставив реальное значение макроса `ABC`. Чему равен конечный результат?
2. Объясните, что такое «область видимости» макроса. До какой строки файла будет действовать макрос `ABC` и как принудительно заставить препроцессор «забыть» этот идентификатор в середине кода?
3. Допустим, в программе объявлен макрос `#define DECREMENT x--`. К какому побочному эффекту приведет его использование в строке `int res = DECREMENT + DECREMENT;`?

### ✏️ Тест к проверочной работе №2
1. **Какой флаг компилятора GCC/Clang необходимо использовать, чтобы остановить сборку после фазы препроцессинга и получить файл `.i`?**
   * А) `-c`
   * Б) `-E`
   * В) `-S`

2. **Видит ли отладчик `GDB` имя макроса `ABC` при построчном выполнении скомпилированного бинарного файла по умолчанию?**
   * А) Да, отладчик всегда видит исходные макросы.
   * Б) Нет, так как для компилятора и отладчика имени `ABC` уже не существует в коде.
   * В) Да, но только если макрос написан внутри функции `main`.

3. **Какое расширение имеет файл промежуточного кода Си, сгенерированный препроцессором?**
   * А) `.o`
   * Б) `.s`
   * В) `.i`

## 💎 Проверочная работа №3: Профессиональный уровень
**Тема**: Сравнительный анализ `#define` и `const`, лексический разбор

### 📋 Задание
Проанализируйте два подхода к объявлению констант в языке Си:
```c
// Вариант 1
#define MAX_SIZE 100

// Вариант 2
const int max_size = 100;
```
1. Проведите детальное сравнение этих вариантов по следующим критериям: наличие строгого контроля типов компилятором, выделение памяти, возможность использования в качестве размера статического массива в старых стандартах Си (C89/C99).
2. Опишите, во что препроцессор превратит следующий синтаксически ошибочный фрагмент кода:
   ```c
   #define TARGET 10;
   int array[TARGET];
   ```
   Какую ошибку выдаст компилятор на этапе лексического анализа результирующей строки?
3. Модифицируйте демонстрационную функцию `demo_macro_substitution_in_main` так, чтобы вместо статического макроса `ABC` использовался макрос-функция `SUM(a, b)`, возвращающая их сумму. Напишите код функции и макроса с защитными круглыми скобками.

### ✏️ Тест к проверочной работе №3
1. **Что такое лексический анализ в контексте компиляции кода, прошедшего препроцессинг?**
   * А) Склеивание объектных файлов в исполняемый бинарник.
   * Б) Разбиение текста программы на элементарные токены (лексемы) констант, ключевых слов и идентификаторов компилятором.
   * В) Удаление комментариев из исходного кода.

2. **Что возвращает демонстрационная функция `demo_macro_substitution_in_main` операционной системе при успешном завершении?**
   * А) Число 7 (значение переменной `z`)
   * Б) Код 0
   * В) Адрес указателя `argv`

3. **Какая директива препроцессора позволяет связать макрос с условием (например, компилировать код только если макрос `ABC` определен)?**
   * А) `#ifdef`
   * Б) `#include`
   * В) `#pragma`

## 🔑 Ответы и критерии оценивания для преподавателя

### Ответы на тестовые задания
* **Тест №1**: 1-Б, 2-А, 3-В
* **Тест №2**: 1-Б, 2-Б, 3-В
* **Тест №3**: 1-Б, 2-Б, 3-А

### Эталоны ответов на практические задания

#### Проверочная работа №1
1. Это означает, что препроцессор работает как обычный текстовый редактор (Find & Replace). Если объявить невалидное выражение вроде `#define ABC 5.5.5`, препроцессор молча подставит его в код: `int y = 5.5.5;`. Ошибка компиляции возникнет позже, когда компилятор попытается распарсить `5.5.5` как числовой токен.
2. Превратится в: `int val = 5 * 2;`.
3. За препроцессинг отвечает утилита `cpp` (C Preprocessor), входящая в пакет компилятора GCC.

#### Проверочная работа №2
1. Шаги: `x = 2`. Переменной `y` присваивается развернутый макрос `5`. Формула: `z = x + y` -> `z = 2 + 5`. Результат: `z = 7`. Функция возвращает `0`.
2. Макросы Си не имеют области видимости функций или блоков `{}`. Объявленный макрос действует глобально от строчки объявления до конца файла. Принудительно отменить его действие можно директивой `#undef ABC`.
3. Строка превратится в `int res = x-- + x--;`. Из-за двукратного изменения переменной `x` в рамках одного выражения происходит нарушение точек следования (Sequence Points), что ведет к неопределенному поведению (`UB`) программы.

#### Проверочная работа №3
1. Сравнение констант:
   * **`#define`**: нет типов, заменяется текстом, не занимает ОЗУ, подходит для объявления размеров статических массивов в C89/C99.
   * **`const`**: строгий контроль типов, занимает память (если адрес запрашивается), в C89/C99 не является константным выражением времени компиляции (нельзя задать размер статического массива).
2. Код развернется в: `int array[10;];`. Компилятор выдаст синтаксическую ошибку (`error: expected ‘]’ before ‘;’ token`), так как точка с запятой из макроса разрушила структуру объявления массива.
3. Модифицированный вариант:
   ```c
   #define SUM(a, b) ((a) + (b))
   int demo_macro_substitution_in_main(int argc, char **argv) {
       int x = 2;
       int y = SUM(x, 5); // Развернется в ((x) + (5))
       int z = x + y;
       return 0;
   }
   ```

* **Хорошо (4)**: Студент правильно рассчитал математический вывод функций, но допустил неточности при сравнении `#define` и `const` или описании нарушения точек следования.
* **Удовлетворительно (3)**: Студент понимает базовый синтаксис языка Си, но путает фазу препроцессинга с фазой компиляции.
