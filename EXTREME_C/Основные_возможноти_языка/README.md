### 🔍 Детальное содержание тем

#### 🛠️ Модуль 1.1–1.3. Препроцессор и метапрограммирование
* **1.1. Директивы препроцессора:** Управление этапом сборки до компиляции. Включение файлов (`#include`). Линейные команды.
* **1.2. Макросы:** Подстановка текста через `#define`. Функциональные макросы. Опасности побочных эффектов. Приоритеты операций.
* **1.3. Условная компиляция:** Ветвление сборки через `#ifdef`, `#ifndef`, `#if`. Защита заголовков (`Include Guards`).

#### 📍 Модуль 1.4–1.9. Указатели и адресная магия
* **1.4. Указатели на переменные:** Переменные, хранящие физические адреса. Прямой доступ к ячейкам памяти.
* **1.5. Синтаксис:** Объявление (`int *p`), взятие адреса (`&x`) и разыменование данных (`*p`). Правила чтения сложных типов.
* **1.6. Арифметические операции:** Смещение указателей на размер типа. Инкремент, декремент и разность адресов.
* **1.7. Обобщенные указатели:** Универсальный тип `void*`. Хранение любых адресов. Необходимость явного приведения типов.
* **1.8. Размер указателей:** Зависимость размера (`sizeof`) от разрядности архитектуры (4 байта для x32, 8 байт для x64).
* **1.9. Висячие указатели:** Указатели на освобожденную или уничтоженную память. Причины падения программ (`Segmentation fault`).

#### 📦 Модуль 1.10–1.15. Анатомия и управление функциями
* **1.10. Общая информация:** Функции как базовые строительные блоки Си. Принцип повторного использования кода.
* **1.11. Анатомия функции:** Сигнатура, прототипы, возвращаемые значения. Спецификаторы аргументов и тело функции.
* **1.12. Роль в архитектуре:** Модульность приложений. Декомпозиция сложных задач. Разделение зон ответственности.
* **1.13. Управление стеком:** Выделение памяти под локальные переменные. Кадр вызова (`Stack Frame`). Опасность переполнения стека.
* **1.14. Передача аргументов:** Копирование данных (`by value`) против передачи адреса (`by reference`) для изменения оригинала.
* **1.15. Указатели на функции:** Хранение адреса исполняемого кода. Реализация динамических вызовов (`callbacks`) и плагинов.

#### 🏛️ Модуль 1.16–1.19. Структурирование данных и кастомные типы
* **1.16. Структуры:** Объединение логически связанных переменных разных типов под одним именем. Сложные объекты.
* **1.17. Зачем нужны структуры:** Избавление от хаоса несвязанных переменных. Моделирование сущностей реального мира.
* **1.18. Зачем нужны пользовательские типы:** Создание читаемых абстракций с помощью `typedef`. Сокращение кодовой базы.
* **1.19. Принцип работы структур:** Расположение полей в памяти. Выравнивание (`alignment`) и физические отступы (`padding`).

#### 🏛️ Модуль 1.20–1.22. Структурирование данных и управление памятью объектов
* **1.20. Структуры (База):** Объединение логически связанных переменных разных типов. Зачем нужны пользовательские типы (`typedef`). Моделирование сущностей реального мира без хаоса в коде.
* **1.20.1. Размещение структур в памяти:** Физическое расположение полей. Концепция выравнивания (`alignment`) и автоматических отступов компилятора (`padding`). Оптимизация размера структуры путем правильной перестановки ее полей. Применение директив паковки (`#pragma pack`).
* **1.21. Вложенные структуры:** Создание сложных, иерархических моделей данных (композиция). Правила доступа к глубоко вложенным полям. Порядок инициализации сложных агрегатных типов.
* **1.22. Указатели на структуры:** Передача объектов в функции без копирования данных. Синтаксис оператора «стрелка» (`->`) против явного разыменования (`(*ptr).field`). Динамическое выделение памяти под структуры на куче.

#### 📁 Лабораторная работа #1: Препроцессор и Строковые макросы
* **Задание:** Реализовать мини-фреймворк для логирования, выводящий сообщения в формате `[ДАТА] [ВРЕМЯ] [ФАЙЛ:СТРОКА] [ФУНКЦИЯ] -> СООБЩЕНИЕ`.
* **Условие:** Логирование должно полностью отключаться на этапе компиляции, если не задан макрос `#define DEBUG`.
* **Критерий приемки:** Нулевой оверхед (code footprint) в релизной сборке. Отсутствие вызовов функций логирования в скомпилированном бинарнике.

#### 🧠 Лабораторная работа #2: Адресная магия и Generic-сортировка
* **Задание:** Написать собственную реализацию алгоритма быстрой сортировки (Analog `qsort`), способную сортировать массив любых типов данных.
* **Условие:** Функция должна принимать `void*` на массив, количество элементов, размер одного элемента и указатель на функцию-компаратор.
* **Критерий приемки:** Запрещено использовать квадратные скобки `[]`. Вся навигация по массиву должна быть написана исключительно на адресной арифметике и явных сдвигах по байтам через `char*`.

#### 💥 Лабораторная работа #3: Исследование Стека и Дамп Памяти
* **Задание:** Написать функцию `dump_stack_frame()`, которая при вызове считывает данные из текущего кадра стека и выводит адреса локальных переменных вызывающей функции.
* **Условие:** Продемонстрировать явный выход за границы массива на стеке (буфер) и перехватить момент, когда висячий указатель начинает портить адрес возврата из функции.
* **Критерий приемки:** Успешная демонстрация (в учебных целях) контролируемого падения программы с выводом "Stack corrupted" до фактического `Segmentation fault`.

#### 🏗️ Лабораторная работа #4: Кастомный сериализатор структур
* **Задание:** Спроектировать структуру данных, описывающую сетевой пакет (заголовок, флаги, контрольная сумма, тело).
* **Условие:** Написать функции, которые упаковывают эту структуру в сырой байтовый буфер для отправки по сети и распаковывают обратно.
* **Критерий приемки:** Размер буфера должен байт-в-байт соответствовать полезным данным. Смещение полей (отступы) должно контролироваться вручную, исключая непредсказуемый padding компилятора.

### 🚀 Мини-проект: `XConfigCore` — Экстремальный парсер конфигураций

#### 📝 Постановка задачи
Необходимо написать модуль, который считывает конфигурационный файл из памяти или диска, динамически строит внутреннюю структуру данных в памяти, позволяет искать значения по ключам с помощью указателей на функции и защищен от любых видов атак через переполнение буфера.

#### 🏗️ Архитектурные требования к проекту

1. **Интерфейс и Инкапсуляция:**
   * Реализовать конфигурационный движок через непрозрачный указатель: `typedef struct xconfig_t xconfig_t;`.
   * Пользователь библиотеки не должен знать, как устроена структура внутри (сокрытие реализации).

2. **Работа с памятью и указателями:**
   * Запрещено выделять фиксированные буферы (например, `char static_buffer[1024]`).
   * Строки должны парситься «на месте» с помощью адресной арифметики и указателей на срезы памяти, либо выделяться динамически строго под размер ключа/значения.
   * Контроль висячих указателей: функция очистки памяти должна принимать указатель на указатель `void xconfig_free(xconfig_t **config)` и занулять его после освобождения ресурсов.

3. **Метапрограммирование и Строгая типизация:**
   * Использовать макросы для безопасного извлечения типов с дефолтными значениями:
     `#define XCONFIG_GET_INT(cfg, key, default_val) ...`
   * Использовать условную компиляцию для вывода детального дерева разбора (AST конфигурации) в `stdout`, если проект собран с флагом `_XCONFIG_VERBOSE`.

4. **Функции и Функциональные указатели:**
   * Реализовать функцию обхода всей конфигурации `xconfig_iterate(xconfig_t *cfg, void (*callback)(const char *key, const char *val, void *user_data), void *user_data)`.

### 🛠️ API Спецификация (Интерфейс модуля)

Вы должны строго следовать данной сигнатуре при написании кода:

```c
#ifndef XCONFIG_H
#define XCONFIG_H

#include <stddef.h>

// Непрозрачная структура (Opaque type)
typedef struct xconfig_t xconfig_t;

// Инициализация и парсинг
xconfig_t* xconfig_load_file(const char *filepath);
xconfig_t* xconfig_load_buffer(const char *buffer, size_t length);

// Безопасное извлечение данных
const char* xconfig_get_string(const xconfig_t *cfg, const char *key, const char *default_value);
int         xconfig_get_int(const xconfig_t *cfg, const char *key, int default_value);

// Итератор по конфигурации
void        xconfig_iterate(const xconfig_t *cfg,
                            void (*callback)(const char *key, const char *val, void *ctx),
                            void *ctx);

// Деструктор (Зануляет указатель)
void        xconfig_free(xconfig_t **cfg);

#endif // XCONFIG_H
```

### 🛑 Критерии жесткой приемки (Hardcore Validation)

* [ ] **Zero Memory Leaks:** Проверка утилитой `valgrind --leak-check=full --show-leak-kinds=all` показывает ровно 0 утерянных байт при любых сценариях (включая битые и поврежденные `.ini` файлы).
* [ ] **Crash Resistance:** Парсер не должен падать в `Segmentation fault` при передаче ему пустых строк, строк без знака `=`, ключей без значений или файлов с поврежденной структурой. При ошибке он обязан корректно возвращать `NULL` или дефолтное значение.
* [ ] **Адресная чистота:** Код парсинга строк написан на указателях (использование функций вроде `strchr`, `strspn` приветствуется, ручной сдвиг указателей обязателен).
* [ ] **Чистота стека:** Отсутствие уязвимостей типа Stack Overflow при парсинге слишком длинных строк конфигурации.

### 🚩 Чек-лист готовности к следующему модулю

Перед тем как двигаться дальше, убедитесь, что вы можете закрыть каждый пункт без подглядывания в шпаргалки:

#### 🛠️ Препроцессор и метапрограммирование
- [ ] Понимаете разницу между стадией препроцессинга и фактической компиляцией кода.
- [ ] Умеете оборачивать макросы в безопасную конструкцию `do { ... } while(0)`.
- [ ] Знаете, как использовать макросы `#ifndef / #define` для защиты от повторного включения заголовков.

#### 📍 Указатели и адресная магия
- [ ] Можете четко объяснить, чем отличается `int *const p` (константный указатель) от `const int *p` (указатель на константу).
- [ ] Понимаете, почему `ptr + 1` смещает адрес не на 1 байт, а на `sizeof(*ptr)` байт.
- [ ] Умеете безопасно приводить `void*` к целевому типу данных перед его разыменованием.
- [ ] Знаете, как зануление указателя после вызова `free()` предотвращает появление висячих указателей (`dangling pointers`).

#### 📦 Функции и стек
- [ ] Можете по шагам расписать, что происходит со стеком (`Stack Frame`) при вызове функции и при возврате из нее.
- [ ] Понимаете, почему категорически нельзя возвращать из функции указатель на ее локальную переменную.
- [ ] Умеете объявлять и вызывать указатель на функцию (например, для передачи его в качестве `callback`).

#### 🏛️ Структуры и типы
- [ ] Понимаете, почему размер структуры `sizeof(struct)` часто больше суммы размеров всех ее полей (концепция `padding`).
- [ ] Умеете использовать `typedef` совместно со структурами для создания чистого и читаемого кода.

#### 🏛️ Структуры и сложные типы данных
- [ ] Понимаете, почему размер структуры `sizeof(struct)` часто больше суммы размеров всех ее полей, и умеете рассчитывать этот оверхед вручную.
- [ ] Умеете использовать директиву паковки структур для сетевых протоколов и понимаете, как это бьет по производительности процессора.
- [ ] Можете без ошибок инициализировать сложную вложенную структуру с помощью назначенных инициализаторов (Designated Initializers).
- [ ] Четко понимаете разницу между передачей структуры в функцию по значению (копирование на стек) и передачей по указателю (`sizeof(void*)`).
